# Edition 1.0 — Standards Backlog

Status: Draft  
Source: PF-11 Design System Audit and PF-12 Engineering Standards Audit  
Reference implementation: PitchIQ

## Purpose

This backlog identifies the first WellTrack Platform Standards that should be drafted from the Platform Audit 1.0 findings.

## P1 standards

| Standard ID | Title | Purpose | Status |
|---|---|---|---|
| WPS-DES-001 | Design Tokens | Define token naming, semantic aliases, motion, z-index and usage rules. | Planned |
| WPS-DES-002 | Components | Define reusable component anatomy, states, token compliance and accessibility expectations. | Planned |
| WPS-DES-003 | Patterns | Define screen/flow pattern ownership and reuse rules. | Planned |
| WPS-ENG-001 | CSS Architecture | Define CSS file structure, token usage, component CSS and lock-file rules. | Planned |
| WPS-ENG-002 | JavaScript Architecture | Define module ownership, route rendering, services and orchestration boundaries. | Planned |
| WPS-REL-001 | Release and Cache Governance | Define GitHub Pages, service worker, cache-busting and release checklist rules. | Planned |
| WPS-QA-001 | Testing and Validation | Define manual smoke tests, device matrix and release gates. | Planned |
| WPS-ACC-001 | Accessibility Baseline | Define accessible names, focus, reduced motion, touch targets and contrast requirements. | Planned |

## Drafting order

1. WPS-REL-001 Release and Cache Governance
2. WPS-QA-001 Testing and Validation
3. WPS-DES-001 Design Tokens
4. WPS-DES-002 Components
5. WPS-DES-003 Patterns
6. WPS-ENG-001 CSS Architecture
7. WPS-ENG-002 JavaScript Architecture
8. WPS-ACC-001 Accessibility Baseline

## Rule

Standards should use normative language:

- MUST
- SHOULD
- MAY
- EXAMPLE

Standards should be practical, short enough to use, and linked to affected catalogues and ADRs.
